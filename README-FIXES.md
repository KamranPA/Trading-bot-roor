# راهنمای اعمال اصلاحات یکپارچه — Trading-bot-roor

## ⚠️ یادآوری مهم
این سیستم **فقط سیگنال تولید و به تلگرام ارسال می‌کند** — هیچ سفارش واقعی
در صرافی ثبت نمی‌شود. تمام اصلاحات زیر همین ماهیت را حفظ کرده‌اند؛ هیچ
منطق ورود/خروج معامله، فرمول امتیازدهی، یا معماری کلی تغییر نکرده — فقط
ناهماهنگی‌ها بین لایو/بک‌تست/ماهانه رفع شده تا هر سه مسیر دقیقاً یک منطق
را اجرا کنند.

## فایل‌ها (جایگزین کن، مسیرها دقیقاً همان مسیر ریپو است)
```
main.py
src/strategy.py
src/backtester.py
src/optimizer.py
src/train_model.py
src/telegram_bot.py
src/database.py
.github/workflows/trade.yml
.github/workflows/backtest.yml
```
همه از نظر نحوی با `python -m py_compile` و YAML با `yaml.safe_load` تست
شده‌اند. فایل‌هایی که دست‌نخورده ماندند (چون مشکلی نداشتند یا محتوایشان را
نداشتم): `config.py`, `src/indicators.py`, `src/strategy_utils.py`,
`src/volume_filter.py`, `src/ai_threshold.py`, `src/csv_store.py`,
`src/coinex_client.py`, `src/brain.py`, `fetcher.py`,
`.github/workflows/monthly-retrain.yml`, `requirements.txt`.

## خلاصه‌ی یکپارچه‌سازی — چرا الان لایو/بک‌تست/ماهانه یکی هستند

| نقطه‌ی ناهماهنگی قبلی | وضعیت جدید |
|---|---|
| `strategy.py` از `AI_THRESHOLD` ثابت سراسری استفاده می‌کرد؛ `backtester.py`/`optimizer.py` از per-symbol کالیبره‌شده | هر سه حالا از `get_ai_threshold(symbol, ...)` استفاده می‌کنند |
| `tp1` در `strategy.py` = `entry ± sl_dist`، در `backtester.py` = `entry ± sl_dist*tp_ratio/2` | هر دو حالا `entry ± sl_dist` (RR=1:1) |
| `optimizer.py` هیچ حداقل طول داده چک نمی‌کرد؛ `backtester.py` حداقل ۲۱۰ ردیف می‌خواست | هر دو حالا `MIN_ROWS_FOR_EVALUATION = 210` |
| `SL_RATIO` در `optimizer.py` همیشه ثابت ۱.۰ پاس داده می‌شد، هیچ‌وقت جست‌وجو نمی‌شد | در `mode="backtest"` واقعاً جست‌وجو می‌شود؛ در `mode="live"` برای سرعت ثابت می‌ماند (مستند‌شده در کد) |
| `train_model.py` فیلتر حجم را **قبل از** ساخت target اعمال می‌کرد → توالی زمانی کندل‌ها می‌شکست → برچسب‌های آموزشی مدل اعوجاج‌دار | حالا target روی سری کامل و پیوسته ساخته می‌شود، فیلتر حجم فقط بعدش برای انتخاب نمونه اعمال می‌شود |
| `trade.yml` هرگز `fetcher.py` را اجرا نمی‌کرد → هر ۵۰ معامله، `optimizer.py` روی `data/4h` خالی اجرا می‌شد → `best_params.json` پاک و commit می‌شد | `fetcher.py` حالا در `trade.yml` هم اجرا می‌شود؛ + لایه‌ی دفاعی در `optimizer.py` که در نبود داده فایل موجود را دست‌نخورده نگه می‌دارد |
| مدل‌های `.pkl` تازه‌آموزش‌دیده در اجرای زنده (`trade.yml`) هرگز commit نمی‌شدند → بازآموزی زنده کاملاً از دست می‌رفت | `trade.yml` حالا `src/models/*.pkl` را هم commit می‌کند |
| `open_positions_count` یک عدد ثابت بود که در طول یک دور اسکن موازی به‌روز نمی‌شد → امکان نقض `MAX_OPEN_POSITIONS` | شمارنده‌ی thread-safe که بلافاصله بعد از ذخیره‌ی هر سیگنال افزایش می‌یابد |
| کالیبراسیون `AI_THRESHOLD` per-symbol گاهی روی test-set خیلی کوچک (حتی ۱۰ نمونه) انجام می‌شد | حداقل ۳۰ نمونه؛ کمتر از آن → fallback امن به مقدار سراسری |
| انتخاب "بهترین پارامتر" در optimizer فقط بر اساس PnL خام بود، بدون حداقل تعداد معامله | حداقل ۱۰ معامله در بازه‌ی ارزیابی لازم است تا یک ترکیب واجد شرایط انتخاب شود |
| heartbeat گاهی دو بار در روز (۲۲:۰۰ و ۲۲:۳۰ UTC) ارسال می‌شد | فقط یک‌بار (`hour==22 and minute<30`) |
| ارسال تلگرام بدون retry — قطعی شبکه = سیگنال گم‌شده | تا ۳ بار retry با backoff |
| `adx_val = feat_adx or adx_score or 0` — وقتی ADX=0 بود، مقدار غلط نمایش داده می‌شد | چک صریح `is not None` |
| `git push --force` در `backtest.yml` — ریسک حذف بی‌صدای commit همزمان از `trade.yml` | `--force-with-lease` |
| هر فراخوانی دیتابیس یک اتصال psycopg2 جدید باز/می‌بست | Connection pool مشترک برای توابع داخلی `database.py` (بدون تغییر رفتار توابع بیرونی مثل `test_db.py`) |

## قدم‌های نصب پیشنهادی
1. هفت فایل پایتون بالا را جایگزین نسخه‌ی فعلی در ریپو کن.
2. دو فایل ورک‌فلو را جایگزین کن.
3. یک اجرای دستی `backtest.yml` (workflow_dispatch) بگیر تا `best_params.json`،
   `ai_thresholds.json` و مدل‌ها یک‌بار از نو، هماهنگ با فرمول‌های اصلاح‌شده،
   بازسازی شوند — چون فرمول TP1/AI_THRESHOLD در بک‌تست کمی تغییر کرده،
   بهتر است قبل از تکیه‌کردن روی گزارش‌های قبلی این را یک‌بار تازه کنی.
4. بعد از آن، `trade.yml` (زمان‌بندی‌شده) و `Monthly Retrain` بدون نیاز به
   کار دستی دیگری، با منطق یکسان ادامه می‌دهند.

## نکته‌ی باقی‌مانده که هنوز نیاز به فایل دارد
`fetcher.py` را برایم نفرستادی — نتوانستم تأیید کنم که در `trade.yml` (هر
۳۰ دقیقه) اجرای آن با API صرافی/Yahoo Finance که استفاده می‌کند از نظر
زمانی/نرخ‌محدودیت مشکلی ایجاد نمی‌کند. اگر می‌خواهی این آخرین حلقه هم
بسته شود، آن را بفرست تا بررسی کنم.
